# oiioii Muse — 技术开发文档

## 1. 产品概述

### 1.1 产品定位

oiioii Muse 是一款**动态灵感拼图生成器**。MVP 阶段以展示网页形态发布，用户可以在网页上直接体验：点击右下角几何桌宠，弹出 IM 风格对话框，通过选择词卡（而非从零打字）的方式，让 AI 读取用户已选词、分析当前创作缺口，并动态生成下一批词卡，逐步拼出故事梗概、角色设定或世界观规则。

后续扩展为浏览器插件，让用户在任意网页上随时召唤 Muse。

核心理念：**选比写容易，拼接产生灵感；用户每选一次，故事就向前长一步。**

### 1.2 MVP 目标

> 验证：用户是否愿意通过"选词组合"的方式获得创意灵感，而不是从零开始构思。

### 1.3 核心使用场景

```text
用户访问 oiioii Muse 展示网页
→ 浏览产品功能介绍和技术架构
→ 点击右下角几何桌宠
→ 弹出 IM 对话框
→ 顶部创作导航显示三条产品路径：[ 故事梗概 ] [ 角色设定 ] [ 世界观规则 ]
→ 用户先选择本次想生成的目标路径
→ 词卡区出现 15-25 个动态词卡，内容由当前路径 + 当前阶段 + 已选词共同决定
→ 用户点选词卡，已选词进入锁定栏（跨路径常驻）
→ 系统更新故事拼图：读取用户已选词，归纳当前创作状态
→ 系统分析缺口：判断当前还缺冲突、动机、规则、代价、真相等哪类元素
→ 系统判断当前路径所处阶段：撒网期 / 拼接期 / 收束期
→ 点击 🔄 换一批时，AI 先读取已选词与缺口，再生成下一批更贴合的词卡
→ 用户可以点「跳出去」临时提高自由词比例，避免故事被过早锁死
→ 用户也可以自己输入词汇加入已选列表
→ 选够 3-20 个词后，点击生成按钮
→ 系统按当前路径输出：故事梗概 / 角色设定卡 / 世界观规则说明
→ 用户可以在对话框里继续修改、追加要求
```

## 2. MVP 功能范围

### 2.1 P0 必做功能

```text
1. 展示网页上线（产品介绍 + 可交互体验）
2. 网页顶部：产品名称 + slogan
3. 网页中间上半部分：产品功能介绍卡片
4. 网页中间下半部分：技术架构介绍卡片
5. 网页右下角：可交互几何桌宠
6. 点击桌宠弹出 IM 对话框
7. 对话框内创作路径导航：[ 故事梗概 ] [ 角色设定 ] [ 世界观规则 ]
8. 路径导航支持点击切换；切换即触发动态词卡重排
9. 词卡区展示 15-25 个大小不等的词汇 / 短语 / 剧情钩子卡

10. 每条路径都有独立三阶段：撒网期 / 拼接期 / 收束期
11. 词卡内容由路径漏斗 + 阶段漏斗 + 反向生词漏斗共同决定
12. 标签来源：本地词库 + AI 动态生成（按路径分别走对应词库与 prompt）
13. 用户点击词卡选中（高亮）
14. 已选词卡进入顶部锁定栏，跨路径常驻，点击 ✕ 可取消
15. 每次刷新前，AI 读取 selectedTags，更新故事拼图并分析当前缺口
16. 🔄 换一批按钮：在当前路径内刷新词卡，不动路径导航与锁定栏
17. 🌪 跳出去按钮：临时提高自由词比例，避免过度收束
18. 用户可手动输入自定义词汇加入已选列表
19. 选词数量限制：最少 3 个，最多 20 个
20. 生成按钮：基于当前路径与已选词输出故事梗概 / 角色设定 / 世界观规则
21. 生成结果以 IM 消息气泡或结构化卡片形式出现在对话框中
22. 用户可在对话框中继续输入修改需求
```

### 2.2 P1 可做功能

```text
1. 桌宠状态动效（idle / thinking / success / error）
2. 桌宠拖拽位置记忆
3. 保存生成的故事到本地
4. 历史对话记录
5. 标签页分类筛选
6. 复制故事梗概
7. 中英文输出切换
```

### 2.3 MVP 暂不做

```text

```

## 3. 技术选型

### 3.1 前端（展示网页）

```text
框架：React 18
开发语言：TypeScript
构建工具：Vite
样式：Tailwind CSS
配色：暗色主题 + 品红色（#FF2D78）点缀
动效：CSS Animation（桌宠呼吸发光、状态切换）
本地存储：localStorage
部署：Vercel / Netlify
```

### 3.2 后端服务

```text
后端框架：FastAPI
开发语言：Python 3.11+
数据校验：Pydantic
HTTP 请求：httpx
服务运行：uvicorn
环境变量：pydantic-settings
部署：Railway / Render / Fly.io
```

### 3.3 技术选择理由

| 模块 | 选择 | 原因 |
|---|---|---|
| React SPA | 前端框架 | 组件化清晰，适合对话框和标签 UI，部署简单 |
| TypeScript | 类型安全 | 降低标签数据和消息结构错误 |
| Vite | 构建工具 | 开发体验好，HMR 快 |
| Tailwind CSS | 样式 | 快速实现暗色主题和标签布局 |
| localStorage | 本地存储 | MVP 不需要账号和数据库 |
| FastAPI | 后端 | 适合 AI 接口代理，开发快 |
| Pydantic | 校验 | 校验 AI 输出 JSON |
| Vercel | 前端部署 | 免费额度够用，自动 CI/CD |

## 6. 动态生词系统设计

### 6.1 总体产品结构：三条创作路径

灵感生成入口从原来的「四方向词云」升级为「三条创作路径」。用户先选择本次想生成的目标，系统再围绕该目标动态生词。

```text
创作路径导航（Creation Path Nav）
  ├─ 故事梗概    Story
  ├─ 角色设定    Character
  └─ 世界观规则  Worldview
        ↓ 决定本轮词卡服务于什么最终产物
动态词卡区（Dynamic Tag Cards）
  ├─ 撒网期：短词、意象词、开放词
  ├─ 拼接期：短语、桥梁词、关系词、线索词
  └─ 收束期：剧情钩子、最终秘密、关键选择、核心规则
```

三条路径的最终输出不同：

| 创作路径 | 关键字段 | 词卡偏向 | 最终输出 |
|---|---|---|---|
| 故事梗概 | `story` | 主角 / 场景 / 事件 / 冲突 / 反转 / 结局钩子 | 200-300 字故事梗概 |
| 角色设定 | `character` | 身份 / 性格 / 创伤 / 欲望 / 秘密 / 关系 / 人物弧光 | 完整角色设定卡 |
| 世界观规则 | `worldview` | 时代 / 核心法则 / 代价 / 禁忌 / 势力 / 世界真相 | 世界观规则说明 |

核心变化：导航栏选择的不只是 UI 分类，而是生成目标的切换。同样的已选词，在不同路径下会被模型生成成不同类型的新词。

```text
已选词：[双重人格] [童年阴影] [镜子房间]

当前路径 = 故事梗概
  → 妹妹留下纸条、每次醒来少一段记忆、镜子里的人替她作案

当前路径 = 角色设定
  → 害怕照镜子、主动遗忘过去、另一个人格保护她、以为自己是凶手

当前路径 = 世界观规则
  → 镜子保存真实人格、记忆可以被人格转移、被遗忘的人会住进镜中
```

### 6.2 动态生词闭环

动态生词系统不是随机词云，而是一个持续更新的创作拼图闭环。

```text
1. 用户选择词卡
   ↓
2. 系统更新故事拼图
   ↓
3. 系统分析当前缺口
   ↓
4. 系统判断当前路径阶段
   ↓
5. 系统生成下一批词卡
   ↓
6. 用户继续选择
   ↓
7. 故事 / 角色 / 世界观逐渐成形
```

每次点击「换一批」时，AI 都要先读取 `selectedTags`，临时归纳当前创作状态，再输出下一批词卡。第一版不强制把每个词永久结构化成复杂数据库零件，但模型输出必须结构化，方便前端渲染和调试。

### 6.3 三层漏斗：路径 + 阶段 + 反向生词

每条路径都共用同一套三层漏斗。

```text
第一层：路径漏斗
  决定词卡服务于故事梗概 / 角色设定 / 世界观规则中的哪一个最终产物。

第二层：阶段漏斗
  决定当前是撒网期 / 拼接期 / 收束期，控制发散与收紧比例。

第三层：反向生词漏斗
  读取所有已选词，分析故事拼图和当前缺口，生成下一批被已选词"染色"的新词卡。
```

路径决定「生成什么」，阶段决定「发散还是收束」，已选词决定「下一批词怎么长出来」。

### 6.4 每条路径的三阶段

#### 故事梗概路径

| 阶段 | 目标 | 词卡范围 | 示例 |
|---|---|---|---|
| 撒网期 | 快速找到故事感觉 | 人物原型 / 场景 / 类型 / 事件 / 氛围 / 冲突苗头 | 失踪妹妹、雨夜车站、时间循环、秘密遗嘱 |
| 拼接期 | 把已选词连接成故事链 | 起因 / 目标 / 阻碍 / 关系 / 线索 / 反转 | 妹妹留下假线索、每次醒来少一段记忆、旧公寓藏着出生记录 |
| 收束期 | 形成可生成梗概的核心结构 | 真相 / 高潮 / 最终选择 / 结局基调 / 悬念钩子 | 妹妹其实从未存在、循环由主角自己制造、雨停后世界重启 |

#### 角色设定路径

| 阶段 | 目标 | 词卡范围 | 示例 |
|---|---|---|---|
| 撒网期 | 打开人物可能性 | 身份 / 职业 / 外貌特征 / 性格 / 社会关系 | 失忆者、私生子、赏金猎人、双面人格 |
| 拼接期 | 让角色产生内部矛盾 | 创伤 / 欲望 / 恐惧 / 秘密 / 弱点 / 关系冲突 | 害怕被抛弃、想找回妹妹、以为自己是凶手 |
| 收束期 | 让角色成为可写的人 | 核心动机 / 隐藏秘密 / 最终选择 / 人物弧光 | 主动遗忘过去、最后选择说出真相、和过去的自己和解 |

#### 世界观规则路径

| 阶段 | 目标 | 词卡范围 | 示例 |
|---|---|---|---|
| 撒网期 | 找到世界的核心想象力 | 时代 / 核心规则 / 异常现象 / 力量体系 / 社会制度 | 记忆交易、梦境入侵、时间税、机械神明 |
| 拼接期 | 让世界规则可运行 | 规则代价 / 使用限制 / 禁忌 / 组织势力 / 阶层影响 | 每次复活失去一段记忆、谎言会变成伤口、死者只能回来一天 |
| 收束期 | 明确世界最重要的法则与冲突源 | 核心法则 / 最终代价 / 世界真相 / 禁忌后果 | 记忆是这个世界的货币、神明其实是旧时代机器、梦境正在吞掉现实 |

### 6.5 阶段判断与比例

阶段判断第一版采用轻规则，后续可升级为「数量 + 结构完整度」。

```text
当前路径下已选词数量 0-1：撒网期
当前路径下已选词数量 2-5：拼接期
当前路径下已选词数量 6+：收束期
```

注意：阶段判断只看当前路径自己的选择数量；语义染色读取所有路径的已选词。

```text
角色路径已选 5 个词，世界观路径已选 1 个词：
  → 角色路径接近收束
  → 世界观路径仍是撒网期

但切到世界观路径时，模型仍然读取角色路径那 5 个词，生成适合这个角色故事的世界观规则。
```

阶段比例：

| 阶段 | 强相关 | 弱相关 | 自由词 | 词卡形态 |
|---|---:|---:|---:|---|
| 撒网期 | 0%-10% | 20%-30% | 60%-80% | 短词、意象词、开放词 |
| 拼接期 | 40%-50% | 30%-40% | 10%-20% | 短语、桥梁词、关系词、线索词 |
| 收束期 | 70%-80% | 10%-20% | 10% | 剧情钩子、最终秘密、关键选择、核心规则 |
| 跳出去 | 20% | 20% | 60% | 临时发散，用于打破套路 |

### 6.6 缺口分析

系统每次刷新前应让模型临时分析当前创作缺口。

故事梗概常见缺口：

```text
主角、场景、引发事件、目标、阻碍、冲突、反转、结局基调
```

角色设定常见缺口：

```text
身份、性格、创伤、欲望、恐惧、秘密、关系、人物弧光
```

世界观规则常见缺口：

```text
时代、核心法则、规则来源、使用代价、禁忌、组织势力、社会影响、世界真相
```

模型返回的 `analysis.missing` 可以用于 UI 提示：

```text
当前缺口：还缺一个推动故事发生的核心冲突。
建议：切到「故事梗概」路径补 1-2 个事件词。
```

### 6.7 词卡生成规则

每次刷新词卡时，固定输出 15-25 个标签。词卡可以随阶段改变密度，但必须避免越到后期越像完整描述句，防止词卡变长、变重、降低用户选择欲。

```text
撒网期：短词
  双胞胎、雨夜、时间循环、废弃医院

拼接期：短语
  被调换的病历、雨夜出现的门、醒来少段记忆

收束期：压缩钩子 / 规则钩子
  自己是凶手、妹妹是人格、复活要替身
```

词卡长度与密度约束：

| 阶段 | 推荐长度 | 最大长度 | 允许形态 | 禁止形态 |
|---|---:|---:|---|---|
| 撒网期 | 2-6 字 | 8 字 | 名词、意象词、短标签 | 完整句子 |
| 拼接期 | 4-10 字 | 12 字 | 短语、关系词、线索词 | 带复杂因果的长句 |
| 收束期 | 4-12 字 | 14 字 | 压缩钩子、真相短语、规则短语 | 15 字以上剧情描述句 |

核心原则：

```text
词卡不是故事句子，而是可选择的灵感零件。

不好：主角发现妹妹其实是另一个人格并决定逃离循环
较好：妹妹是人格
较好：决定逃离循环
较好：打碎最后镜子
```

如果模型需要表达复杂信息，应拆成多个短词卡，而不是塞进一个长词卡。

```text
长句：主角发现每次复活都会替换掉一个亲人的记忆
拆分：复活代价、替换记忆、亲人遗忘
```

前端展示规则：

- 词卡默认只展示一行，最多两行。
- 超过推荐长度的词卡应由后端压缩后再返回。
- 如仍超过 UI 宽度，前端可省略显示，但 hover / 点击详情中展示完整含义。
- 收束期可以更具体，但仍必须保持「可点选」「可拼接」「一眼能懂」。

首批进入对话框时，默认路径为「故事梗概」，可从本地词库抽取，零 AI 延迟；用户切换路径或点击「换一批」之后，调用 AI 动态生成。

### 6.8 路径切换与已选词联动

切换路径时词卡立即重排，已选词常驻锁定栏，并反向影响新路径的词卡内容。

```text
锁定栏：[双重人格] [童年阴影] [镜子房间]
切到「世界观规则」
  → 后端收到 path=worldview + selectedTags=[双重人格, 童年阴影, 镜子房间]
  → 返回偏向"适合这个心理悬疑故事"的世界观规则词：
     镜子保存真实人格、记忆可以被人格转移、被遗忘的人会住进镜中
```

### 6.9 标签数据结构

第一版采用轻结构化：输入保持简单，输出保持规整。

```ts
type CreationPath = 'story' | 'character' | 'worldview';

type FunnelStage = 'spread' | 'stitch' | 'narrow';

interface Tag {
  id: string;
  text: string;
  path: CreationPath;          // 该词卡服务的创作路径
  source: 'local' | 'ai' | 'user';
  stage?: FunnelStage;         // 该词卡来自哪个阶段
  isCrossover?: boolean;       // 保留字段：后续可用于跨路径提示
}

interface DynamicTagAnalysis {
  storySeed: string;           // 当前已选词形成的创作种子
  stage: FunnelStage;          // 当前路径阶段
  currentGoal: string;         // 当前路径本轮词卡目标
  missing: string[];           // 当前缺口
  tone: string;                // 当前整体调性
  reason?: string;             // 为什么这样推荐
}
```

后续如果需要更强控制，再增加 `slot` / `semanticRole` / `toneTags` / `relation` 等字段。

## 10. AI Prompt 设计

### 10.1 动态词卡生成（按创作路径路由）

每条创作路径走独立 prompt 模板，输入参数：当前路径、已选词、当前路径阶段、是否跳出去、需要生成的数量。

通用要求：

```text
1. 先理解 selectedTags 正在形成什么创作种子
2. 判断当前 path 处于撒网期 / 拼接期 / 收束期
3. 分析当前缺口：故事缺冲突、角色缺动机、世界观缺代价等
4. 根据 path + stage + missing 生成下一批词卡
5. 不要重复 selectedTags 和 excludeTexts 中的任何词
6. 词卡要短、有画面、有选择欲；收束期允许更具体，但仍必须是短词卡，不要变成完整描述句
7. 严格遵守长度上限：撒网期≤8字，拼接期≤12字，收束期≤14字；超过上限时必须压缩或拆分
8. 严格返回 JSON：analysis + tags
```

三路径词性偏向：

| 路径 | Prompt 词性引导 |
|---|---|
| story | 突出"故事会怎么发生"——主角、场景、事件、目标、阻碍、反转、结局钩子 |
| character | 突出"这个人为什么值得写"——身份、性格、创伤、欲望、恐惧、秘密、关系、人物弧光 |
| worldview | 突出"这个世界如何运转"——核心规则、代价、禁忌、制度、势力、社会影响、世界真相 |

阶段 Prompt 重点：

```text
撒网期：打开可能性，多给短词、意象词、开放词，不要太早替用户定型。
拼接期：连接已有词，多给桥梁词、关系词、线索词、规则代价，让词与词之间产生因果。
收束期：帮助成形，多给真相、反转、最终选择、人物弧光、核心法则、结局钩子。
跳出去：临时提高自由词比例，打破当前套路。
```

推荐输出格式：

```json
{
  "analysis": {
    "storySeed": "一个有双重人格和童年阴影的人，被镜子房间困住。",
    "stage": "stitch",
    "currentGoal": "补充角色的动机、恐惧和隐藏秘密。",
    "missing": ["核心欲望", "最大恐惧", "隐藏秘密"],
    "tone": "心理悬疑、阴郁、压抑",
    "reason": "当前已有角色创伤和关键场景，但角色还缺少行动动机。"
  },
  "tags": [
    { "text": "害怕镜子", "path": "character", "stage": "stitch" },
    { "text": "自认凶手", "path": "character", "stage": "stitch" },
    { "text": "人格保护她", "path": "character", "stage": "stitch" }
  ]
}
```

### 10.2 生成故事梗概

```text
要求：
1. 基于用户选择的词汇组合，生成一段故事情节梗概
2. selectedTags 中每条带有 path，按创作路径归类阅读：
   - character 路径的词作为主角设定
   - worldview 路径的词作为背景规则
   - story 路径的词作为场景、冲突、事件与情节钩子
3. 故事中每一类词都要在文本里"看得见痕迹"，让用户感觉"这是我抽出来的故事"
4. 字数控制在 200-300 字
5. 要有画面感，像在描述一个电影片段
6. 有明确的主角、场景、冲突
7. 结尾留有悬念或余韵，不要写完整结局
8. 语言风格：简洁、有节奏感、小学生也能读懂
9. 不要解释创作思路，直接输出故事
```

### 10.3 生成角色设定

```text
要求：
1. 基于用户选择的词汇组合，生成一张完整角色设定卡
2. selectedTags 中 character 路径的词优先作为角色本体
3. story 路径的词可作为角色所处事件、场景或命运背景
4. worldview 路径的词可作为角色所属世界规则或能力限制
5. 输出应包含：暂称、身份、外在性格、内在创伤、核心欲望、最大恐惧、隐藏秘密、关系冲突、人物弧光、可用剧情钩子
6. 语言简洁、有画面感，不要写成论文式设定
7. 直接输出角色设定，不要解释创作思路
```

### 10.4 生成世界观规则

```text
要求：
1. 基于用户选择的词汇组合，生成一套可用于故事创作的世界观规则
2. selectedTags 中 worldview 路径的词优先作为核心规则来源
3. character 路径的词用于判断这个世界如何影响具体人物
4. story 路径的词用于判断这个世界适合承载什么事件和冲突
5. 输出应包含：世界暂称、核心规则、规则来源、使用代价、禁忌、社会影响、主要势力、潜在冲突、故事钩子
6. 世界观必须有代价、有漏洞、有冲突空间，不能只是背景装饰
7. 直接输出世界观规则，不要解释创作思路
```

### 10.5 修改 / 续写结果

```text
要求：
1. 基于原始故事和用户的修改需求，输出修改后的完整故事
2. 保持 200-300 字
3. 尊重用户的修改意图
4. 保持故事的连贯性和画面感
5. 直接输出修改后的故事，不要解释改了什么
```

## 12. 安全与成本控制

### 12.1 API Key 安全

AI API Key 只存在 FastAPI 后端环境变量中，不写入前端代码。

### 12.2 鉴权

前端请求后端时带 `X-App-Token` 头，后端校验通过后才调用 AI。

### 12.3 限流

```text
每个 IP 每分钟最多 10 次 AI 请求
每个 IP 每天最多 50 次 AI 请求
```

### 12.4 成本优化

- 首页标签全部走本地词库，零 AI 成本
- AI 推荐标签用轻量模型（如 gpt-4o-mini）
- 故事生成限制 300 字输出，控制 token 消耗

## 16. 技术总结

```text
React SPA 展示网页（Vite + TypeScript + Tailwind）
→ 产品介绍 + 可交互体验一体化
→ 右下角几何桌宠 + IM 对话框
→ 三路径创作导航：[ 故事梗概 ] [ 角色设定 ] [ 世界观规则 ] + 动态词卡区
→ 动态词卡 = 当前路径 + 当前阶段 + 已选词反向染色 + 缺口分析
→ 每条路径独立拥有撒网期 / 拼接期 / 收束期
→ 已选词跨路径常驻锁定栏，反向影响新路径的词卡内容
→ FastAPI 后端按 path 路由到独立词库与 prompt 链路
→ 用户选词组合 → AI 更新故事拼图 → 分析缺口 → 生成下一批词卡
→ 按当前路径生成故事梗概 / 角色设定 / 世界观规则
→ 对话框内完成修改迭代
```

一句话：

> oiioii Muse 是一个"选词拼接灵感"的动态创作生成器——用户先选择 [ 故事梗概 ] [ 角色设定 ] [ 世界观规则 ] 中的创作目标，再通过词卡持续选择；Muse 每次刷新都会读取已选词、分析当前缺口、生成下一批动态词卡，让故事、角色或世界观逐步成形。MVP 以展示网页形态发布，让用户在线即可体验。
