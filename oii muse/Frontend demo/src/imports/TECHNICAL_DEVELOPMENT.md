# oiioii Muse — 技术开发文档

## 1. 产品概述

### 1.1 产品定位

oiioii Muse 是一款**灵感快速生成器**。MVP 阶段以展示网页形态发布，用户可以在网页上直接体验：点击右下角几何桌宠，弹出 IM 风格对话框，通过选词（而非打字）的方式快速组合灵感元素，一键生成 200-300 字的故事情节梗概。

后续扩展为浏览器插件，让用户在任意网页上随时召唤 Muse。

核心理念：**选比写容易，组合产生惊喜。**

### 1.2 MVP 目标

> 验证：用户是否愿意通过"选词组合"的方式获得创意灵感，而不是从零开始构思。

### 1.3 核心使用场景

```text
用户访问 oiioii Muse 展示网页
→ 浏览产品功能介绍和技术架构
→ 点击右下角几何桌宠
→ 弹出 IM 对话框，里面是大小不等的词汇标签
→ 用户选择感兴趣的词（地点、角色、情节、时间、类型等）
→ 可以刷新换一页新标签，继续选
→ 也可以自己输入词汇加入
→ 选够 3-20 个词后，点击生成按钮
→ 对话框里以聊天消息形式出现一段故事梗概（200-300 字）
→ 用户可以在对话框里继续修改、追加要求
```

## 2. MVP 功能范围

### 2.1 P0 必做功能

```text
1. 展示网页上线（产品介绍 + 可交互体验）
2. 网页顶部：产品名称 + slogan
3. 网页中间上半部分：产品功能介绍卡片
4. 网页中间下半部分：技术架构介绍卡片
5. 网页右下角：可交互几何桌宠
6. 点击桌宠弹出 IM 对话框
7. 对话框内展示词汇标签页（大小不等的矩形标签）
8. 标签分类：地点、角色性格、情节、时间跨度、故事类型等
9. 标签来源：一半本地词库 + 一半 AI 动态生成
10. 用户点击标签选中（高亮）
11. 已选标签在对话框上方区域高亮展示
12. 刷新按钮：换一页新标签，已选标签保留
13. 用户可手动输入自定义词汇加入已选列表
14. 选词数量限制：最少 3 个，最多 20 个
15. 生成按钮：基于已选词汇生成故事梗概
16. 故事梗概以 IM 消息气泡形式出现在对话框中
17. 用户可在对话框中继续输入修改需求
```

### 2.2 P1 可做功能

```text
1. 桌宠状态动效（idle / thinking / success / error）
2. 桌宠拖拽位置记忆
3. 保存生成的故事到本地
4. 历史对话记录
5. 标签页分类筛选
6. 复制故事梗概
7. 中英文输出切换
```

### 2.3 MVP 暂不做

```text
1. Chrome 浏览器插件版本
2. Tauri 桌面端
3. 用户账号系统
4. 云同步
5. 支付系统
6. 桌宠换装和养成
```

## 3. 技术选型

### 3.1 前端（展示网页）

```text
框架：React 18
开发语言：TypeScript
构建工具：Vite
样式：Tailwind CSS
配色：暗色主题 + 品红色（#FF2D78）点缀
动效：CSS Animation（桌宠呼吸发光、状态切换）
本地存储：localStorage
部署：Vercel / Netlify
```

### 3.2 后端服务

```text
后端框架：FastAPI
开发语言：Python 3.11+
数据校验：Pydantic
HTTP 请求：httpx
服务运行：uvicorn
环境变量：pydantic-settings
部署：Railway / Render / Fly.io
```

### 3.3 技术选择理由

| 模块 | 选择 | 原因 |
|---|---|---|
| React SPA | 前端框架 | 组件化清晰，适合对话框和标签 UI，部署简单 |
| TypeScript | 类型安全 | 降低标签数据和消息结构错误 |
| Vite | 构建工具 | 开发体验好，HMR 快 |
| Tailwind CSS | 样式 | 快速实现暗色主题和标签布局 |
| localStorage | 本地存储 | MVP 不需要账号和数据库 |
| FastAPI | 后端 | 适合 AI 接口代理，开发快 |
| Pydantic | 校验 | 校验 AI 输出 JSON |
| Vercel | 前端部署 | 免费额度够用，自动 CI/CD |

## 6. 标签系统设计

### 6.1 标签分类

```text
- 地点：废墟、深海、空间站、老巷子、游乐场...
- 角色性格：孤僻、热血、阴郁、天真、偏执...
- 情节元素：背叛、重逢、逃离、献祭、觉醒...
- 时间跨度：一瞬间、三天、十年、轮回、末日倒计时...
- 故事类型：悬疑、科幻、治愈、荒诞、公路片...
- 视觉元素：雨夜、霓虹、雾气、镜子、火焰...
- 情绪氛围：压抑、释然、癫狂、宁静、紧迫...
```

### 6.2 标签生成策略

每页展示 12-16 个标签：
- 6-8 个来自本地词库（随机抽取，保证分类覆盖）
- 6-8 个由 AI 动态生成（基于当前已选标签做关联推荐）

首页全部从本地词库抽取（无需等待 AI），刷新后的页面混合 AI 生成。

### 6.3 标签数据结构

```ts
interface Tag {
  id: string;
  text: string;
  category: TagCategory;
  source: 'local' | 'ai';
}

type TagCategory =
  | 'location'
  | 'character'
  | 'plot'
  | 'timespan'
  | 'genre'
  | 'visual'
  | 'mood';
```

## 10. AI Prompt 设计

### 10.1 生成推荐标签

```text
要求：
1. 基于用户已选的标签，推荐关联但有新意的词汇
2. 覆盖不同分类（地点、角色、情节、时间、类型、视觉、情绪）
3. 每个词 2-4 个字，简洁有画面感
4. 不要重复已选标签
5. 严格返回 JSON
```

### 10.2 生成故事梗概

```text
要求：
1. 基于用户选择的词汇组合，生成一段故事情节梗概
2. 字数控制在 200-300 字
3. 要有画面感，像在描述一个电影片段
4. 有明确的主角、场景、冲突
5. 结尾留有悬念或余韵，不要写完整结局
6. 语言风格：简洁、有节奏感、小学生也能读懂
7. 不要解释创作思路，直接输出故事
```

### 10.3 修改 / 续写故事

```text
要求：
1. 基于原始故事和用户的修改需求，输出修改后的完整故事
2. 保持 200-300 字
3. 尊重用户的修改意图
4. 保持故事的连贯性和画面感
5. 直接输出修改后的故事，不要解释改了什么
```

## 12. 安全与成本控制

### 12.1 API Key 安全

AI API Key 只存在 FastAPI 后端环境变量中，不写入前端代码。

### 12.2 鉴权

前端请求后端时带 `X-App-Token` 头，后端校验通过后才调用 AI。

### 12.3 限流

```text
每个 IP 每分钟最多 10 次 AI 请求
每个 IP 每天最多 50 次 AI 请求
```

### 12.4 成本优化

- 首页标签全部走本地词库，零 AI 成本
- AI 推荐标签用轻量模型（如 gpt-4o-mini）
- 故事生成限制 300 字输出，控制 token 消耗
